# E2E Tests (Playwright)

Playwright-driven end-to-end tests in `e2e/` — real browser against the
real `vera-demo` Supabase project. Asserts user-visible behavior through
the full stack (Vercel → Kong → PostgREST → Postgres → RLS).

For deeper architectural context, see
[../architecture/e2e-testing-primer.md](../architecture/e2e-testing-primer.md).
For the **why** of specific patterns, see
[target-test-architecture.md](target-test-architecture.md).

---

## Commands

```bash
npm run e2e                            # full suite, headless
npm run e2e -- --headed                # watch in a real browser
npm run e2e -- --grep "Login"          # filter by spec/test name
npm run e2e -- --workers=1             # single worker (debugging)
npm run e2e -- --project=chromium      # one browser only
npm run e2e:report                     # open HTML report from last run
npm run e2e:excel                      # generate xlsx report
```

First-time setup: `npx playwright install` (downloads browser binaries).

## Page Object Model (PoM)

Every major page has a PoM in [`e2e/poms/`](../../e2e/poms/) that wraps
selectors and common interactions. **Specs talk to PoMs, never directly
to selectors.** This means a UI refactor breaks one PoM file, not 30
specs.

23 PoMs at the time of writing — full list in
[`e2e/poms/`](../../e2e/poms/). Examples:

- `LoginPom.ts` — sign-in flows.
- `AdminShellPom.ts` — admin sidebar nav, common assertions.
- `JuryEvalPom.ts` — score writing, group nav.
- `WizardPom.ts` — setup wizard step navigation.

When adding a new admin page, write a PoM first; specs come second.

## Fixtures

Static test data lives in [`e2e/fixtures/`](../../e2e/fixtures/):

- `seed-ids.ts` — UUIDs from the deterministic demo seed. Specs that
  need to reference a known org / period / juror import from here.
- `projects-sample.csv` — CSV import test fixture.

The demo seed is generated by
[`scripts/generate_demo_seed.js`](../../scripts/generate_demo_seed.js)
and is deterministic (same seed = same UUIDs across runs). See
[../operations/demo-environment.md](../operations/demo-environment.md).

## Auth in E2E

Three identities used by the suite:

| Identity | Source | Used for |
| --- | --- | --- |
| Demo super-admin | `demo-admin@vera-eval.app` (no password — auto-login env vars) | Most admin specs |
| Demo tenant-admin | `tenant-admin@vera-eval.app` + bcrypt password from seed | Tenant isolation specs |
| Juror sessions | Server-issued tokens via entry-token flow | Jury specs |

The sign-in helper lives in [`e2e/helpers/`](../../e2e/helpers/) — never
inline `page.fill('input[name=email]', ...)` in a spec.

## Skip policy

Skipped tests must include a reason comment and a tracking issue or
plan reference.

```typescript
test.skip("score-edit unlock flow", async ({ page }) => {
  // SKIP: rpc_admin_unlock_score not implemented yet.
  // Tracked in premium-saas-test-upgrade-plan.md § 5.
});
```

`.skip()` without context rots silently; the audit at
[e2e-security-skip-audit.md](e2e-security-skip-audit.md) reviews skips
and either reactivates or formally excludes them.

## Reporting

Playwright produces three artifacts after a run:

- `test-results/playwright-report/` — interactive HTML report
  (`npm run e2e:report` to open).
- `test-results/playwright-results.json` — machine-readable results;
  parsed by CI to fail on regressions.
- xlsx report — `npm run e2e:excel` generates an xlsx summary, useful
  for sharing with non-technical stakeholders.

Allure reporter slot exists (`test-results/allure-results`) but is not
yet wired up — tracked in
[premium-saas-test-upgrade-plan.md](premium-saas-test-upgrade-plan.md).

## Concurrency model

Default: parallel workers, sharded in CI.

Some specs cannot run in parallel:

- **Maintenance-mode E2E** runs in a dedicated CI job after admin specs
  complete (see commits `a4a6d3b6`, `a852cb0d`).
- **Tenant isolation specs** assume specific seeded UUIDs and cannot
  share a worker with specs that mutate the same entities.

If a spec needs serial execution, mark it: `test.describe.serial(...)`.
If two specs collide, document why and either move one to a different
file or add a serial marker.

## Anti-patterns

- **Selector inline in spec.** Always extract to PoM. The 11-file
  helper rewrite mid-2026 was directly caused by selectors scattered
  across specs.
- **Fixed sleeps** (`await page.waitForTimeout(1000)`). Use
  `await expect(...).toBeVisible({ timeout: ... })` or `waitFor`.
- **Hardcoded URLs.** Use `playwright.config.ts` baseURL; specs use
  relative paths.
- **Direct DB writes from a spec.** Use the application's RPCs. If you
  need to set up state that the UI cannot reach (e.g., simulating an
  expired session), do it in a fixture or `global.setup.ts`.
- **Snapshot bloat.** Visual specs in `e2e/visual/` have a per-page
  snapshot. Don't over-snapshot — text-content assertions usually
  suffice.

## Drift sentinels related to E2E

- `data-testid` ownership: testids are E2E's territory. When a unit
  test needs a new testid, coordinate (don't add silently).
- `npm run check:rls-tests` — every isolated table has a paired pgTAP
  test, supplementing E2E security specs.
- The `e2e/security/tenant-isolation.spec.ts` file is the canonical
  cross-tenant smoke check; if RLS regresses, this spec fails first.
