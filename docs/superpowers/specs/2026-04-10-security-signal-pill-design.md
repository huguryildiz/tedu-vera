# Security Signal Pill — Data-Driven Replacement for Hardcoded "Secure" Badge

**Date:** 2026-04-10
**Status:** Approved
**Scope:** `src/admin/pages/SettingsPage.jsx`, new `SecuritySignalPill` component, new `utils/computeSecuritySignal.js` helper

## Problem

The Security & Sessions card in admin Settings has a hardcoded green "Secure" pill in the top-right. It is a purely cosmetic badge — it has no conditional logic, never reflects actual account state, and gives the user no information they could not already read from the card itself. The badge is identical whether the account has 1 clean session or 6 sessions across 3 countries with a 90-day-old login.

The surrounding card already surfaces rich data: `last_sign_in_at`, `adminSessions.length`, `authMethod`, and a per-session drawer with `ip_address`, `country_code`, `last_activity_at`, `expires_at`. None of that feeds the pill.

The rest of VERA follows an "active assistant" design language — analytics insight banners, top projects highlights, and KPI strip dynamics all compute state from real data and tell the user what it means. A static "Secure" pill in the middle of that breaks the tone and wastes a valuable surface.

## Solution

Replace the static badge with a **Security Signal Pill**: a three-state indicator computed from the data already in the card, paired with a click-to-open "verdict-first" popover that explains *why* the account sits at its current state and links to the relevant action.

### Pill states

Three levels, tied to a max-severity rollup of four signals:

| Level | Label | Color | Meaning |
|---|---|---|---|
| `secure` | `Secure` | green (`badge-success`) | All four signals are ok |
| `review` | `Review` | amber (`badge-warning`) | At least one signal is `warn`, no signals are `bad` |
| `risk` | `At Risk` | red (`badge-danger`) | At least one signal is `bad` |

Pill layout: dot + label + down-chevron (`ChevronDown` from lucide-react, `10px`, `opacity: 0.6`). The chevron signals that the pill is clickable. When the state is `secure` the chevron still appears so the pill stays visually consistent across states.

### Signals (v1)

Four signals feed the rollup. Each returns `ok`, `warn`, or `bad`. All data is already loaded — no new API calls, no new RPCs, no DB migration.

| Signal | Source | `ok` | `warn` | `bad` |
|---|---|---|---|---|
| `sessionCount` | `adminSessions.length` | ≤ 2 | 3 or 4 | ≥ 5 |
| `countryDiversity` | distinct non-null `country_code` across `adminSessions` | 1 (or 0 when unknown) | 2 | ≥ 3 |
| `lastLoginFreshness` | `session.user.last_sign_in_at` | ≤ 14 days | 15–45 days | > 45 days |
| `expiredSessions` | `adminSessions.filter(s => Date.parse(s.expires_at) < Date.now()).length` | 0 | 1 | ≥ 2 |

**Severity rollup:** `bad > warn > ok`. The pill state is the worst severity across all four signals. A single `bad` → pill is `At Risk`, even if the other three are `ok`.

**Unknown handling:** If `adminSessions` is still loading, the pill renders a neutral loading state (grey dot + `—`) and is not clickable. If `adminSessions` finished loading but is empty (no tracked sessions yet), we fall back to treating the current Supabase session as 1 session and only evaluate `lastLoginFreshness`; the pill still reflects that result.

### Verdict sentence generation

The popover opens with a one-sentence plain-language verdict. It is generated by scanning signals in severity order (`bad` first, then `warn`) and templating the top 1–2 non-ok signals into a phrase.

Verdict phrasing rules:

- **All ok (`secure`)** → verdict title and reason are `null`. The popover still opens when clicked and still shows the factor list and footer action, but it renders **without** the verdict banner. The header area is replaced by a single neutral line: `"All security signals are clear."` in muted text with a small green check icon.
- **One warn / one bad** → title is imperative (`"This account needs a review."` for review, `"This account is at risk."` for risk). Reason is `"{Signal phrase} pushed this account to {state}."` Example: `"4 active sessions pushed this account to Review."`
- **Two or more warn / bad** → same title rule. Reason is `"{Top signal A} and {top signal B} pushed this account to {state}."` Example: `"4 active sessions and 21 days of inactivity pushed this account to Review."`
- **Three or more** → same template as two; the extras live in the factor list below. Never list more than two in the sentence.

Signal → phrase map (used for verdict and factor labels):

```text
sessionCount   → "{N} active sessions"   (warn/bad only; ok omitted)
countryDiversity → "{N} countries"
lastLoginFreshness → "{N} days of inactivity"
expiredSessions → "{N} expired sessions"
```

### Popover structure

Click the pill → popover opens anchored to the pill's bottom-right. Click outside, press `Escape`, or click the pill again → closes.

Three sections, top to bottom:

**1. Verdict banner** (only when state is `review` or `risk`)

- Background: matches state (`#fef9c3` for review, `rgba(225,29,72,0.08)` for risk)
- Top border arrow matches banner background
- Title line: short imperative (`"This account needs a review."` / `"This account is at risk."`)
- Reason line: the generated verdict sentence, one font size smaller, muted
- When state is `secure`: no banner, no arrow tint. Instead a compact header row inside the popover body shows a green check icon + `"All security signals are clear."` in muted text, with a divider below it.

**2. Factor list**

All four factors rendered compact, one per row. Each row:

- Left: factor label (`"Active sessions"`, `"Countries"`, `"Last login"`, `"Expired sessions"`)
- Right: value pill (`"4 · high"`, `"1 · ok"`, `"21d · stale"`, `"0 · ok"`) — monospace font, background color matches signal severity

Factors are always shown in the same order regardless of severity, so users build spatial familiarity.

**3. Footer action**

A single link-style button: `"Review sessions →"` (always, even when `secure`). Clicking it:

- Closes the popover
- Calls `setViewSessionsOpen(true)` — opens the existing `ViewSessionsDrawer`

This is the "aksiyon mesafesi sıfır" payoff — the pill tells the user something is off, and the same popover gives them the one-click path to act on it.

### Component boundaries

**New file:** `src/admin/utils/computeSecuritySignal.js`

Pure function, no React, no Supabase calls:

```text
computeSecuritySignal({ adminSessions, lastLoginAt, loading }) → {
  state: "loading" | "secure" | "review" | "risk",
  signals: {
    sessionCount: { value: number, severity: "ok" | "warn" | "bad" },
    countryDiversity: { value: number, severity: "ok" | "warn" | "bad" },
    lastLoginFreshness: { value: number, severity: "ok" | "warn" | "bad" },
    expiredSessions: { value: number, severity: "ok" | "warn" | "bad" },
  },
  verdict: {
    title: string | null,     // "This account needs a review."
    reason: string | null,    // "4 active sessions and 21 days of inactivity pushed this account to Review."
  }
}
```

Fully unit-testable — takes plain objects, returns plain objects. All threshold constants live here as named exports so tests can assert them directly.

**New file:** `src/admin/components/SecuritySignalPill.jsx`

Presentational React component. Props:

```text
<SecuritySignalPill
  signal={signalResult}       // output of computeSecuritySignal
  onReviewSessions={() => void}  // opens ViewSessionsDrawer
/>
```

Owns: pill button, popover open state, outside-click / `Escape` handling, keyboard a11y (`aria-expanded`, `aria-controls`), and all visual rendering. Does not own data loading or the drawer itself.

**`SettingsPage.jsx` changes:**

- Import `computeSecuritySignal` and `SecuritySignalPill`
- Compute `securitySignal` via `useMemo` based on `adminSessions`, `adminSessionsLoading`, and `lastLoginAt`
- Replace the hardcoded `<span className="badge badge-success">Secure</span>` with `<SecuritySignalPill signal={securitySignal} onReviewSessions={() => setViewSessionsOpen(true)} />`
- No other changes to the card — the three KPI cells (Last Login / Sessions / Auth Method) and the button row stay exactly as they are

## Visual details

- **Pill sizing**: matches existing `.badge` pattern (small, rounded-full, 11px font). Adds `cursor: pointer`, `display: inline-flex`, `gap: 6px`, and a subtle hover lift (`box-shadow: 0 1px 2px rgba(0,0,0,0.04)` on hover) to signal clickability without being noisy.
- **Popover width**: `300px` fixed. Responsive fallback: on viewports narrower than 420px, the popover becomes `calc(100vw - 28px)` and anchors to the right edge of the card instead of the pill.
- **Popover shadow**: `0 10px 30px -10px rgba(15,23,42,0.25), 0 2px 6px rgba(15,23,42,0.05)` — matches the drawer/modal depth used elsewhere in Settings.
- **Popover arrow**: 10×10 rotated square pinned to the top-right, background and border color match the banner (or `#fff` + border when `secure` has no banner).
- **Dark mode**: all color tokens come from existing CSS variables (`--success`, `--warning`, `--danger`, `--surface-1`, `--border`). The popover background uses `--surface-elevated` / equivalent so it sits correctly on both themes. Factor value pills use the same light-mode contrast overrides pattern as `fb-alert`.
- **Icons**: only `ChevronDown` from lucide-react for the pill; factor severity chips use text tags (`"ok"`, `"high"`, `"stale"`) instead of icons to keep the popover compact.

## Accessibility

- Pill is a real `<button>` with `aria-expanded`, `aria-controls`, and a descriptive `aria-label` (`"Security signal: Review. Click to see details."`).
- Popover has `role="dialog"`, `aria-labelledby` pointing to the verdict title (or a visually-hidden label when `secure`).
- Focus trap is not needed (popover is lightweight, not modal), but `Escape` must close it and return focus to the pill.
- Outside-click detection uses a single document-level listener mounted only while the popover is open.
- The footer "Review sessions →" is a real focusable `<button>`, not a styled `<span>`.

## Testing

Two test files plus manual verification, all using the existing `qaTest` pattern.

**`src/admin/__tests__/computeSecuritySignal.test.js`** — unit tests for the pure function:

- All four signals at `ok` → state `secure`, verdict null
- One signal at `warn` → state `review`, verdict single-signal template
- Two signals at `warn` → state `review`, verdict two-signal template
- One signal at `bad` → state `risk`, even if others are ok
- `loading: true` → state `loading`, all signal severities `ok` (not evaluated)
- Each threshold boundary (2→3 for sessionCount, 14→15 for lastLogin, etc.)
- `adminSessions` empty but lastLoginAt fresh → state `secure` with sessionCount=1
- All sessions with null `country_code` → countryDiversity value 0, severity `ok`

**`src/admin/__tests__/SecuritySignalPill.test.jsx`** — component tests:

- Renders correct label for each state
- Popover closed by default
- Clicking pill opens popover
- Clicking pill again closes popover
- Pressing Escape closes popover
- Clicking outside closes popover
- Banner absent when state is `secure`, present when `review` or `risk`
- Clicking "Review sessions" calls `onReviewSessions` and closes popover
- `aria-expanded` toggles correctly

**Integration wiring** — `SettingsPage` has no existing integration test file and would require mocking `AuthProvider`, tenant membership, profile loading, and Supabase. The wiring introduced in this spec is 3 lines (import, `useMemo`, one JSX swap) and is covered by manual verification in the live app. Adding a full integration test is explicitly out of scope.

## Out of scope

The following are intentionally deferred and not part of this spec:

- **MFA status signal** — requires new Supabase Auth queries and backend work. Worth having but belongs in a follow-up spec.
- **Password age signal** — requires storing `password_changed_at` (not currently tracked). Follow-up.
- **Per-session risk scoring** — showing a per-session risk tag inside `ViewSessionsDrawer`. Different surface, different spec.
- **Historical trend** — e.g. "risk has improved since last week". Requires a time series that does not exist yet.
- **Email/notification on state change** — waking up the admin when their account transitions to `At Risk`. Needs the v2.0 email infra.
- **Organization-level rollup** — showing the super admin a "review needed across 3 admins" summary on the Overview tab. Different surface.

## Rollout

Single PR. No feature flag needed — the pill is purely additive on one card, the rest of Settings is untouched, and a regression is easy to spot visually. Tests must pass, `npm run check:no-native-select` must pass, and the light/dark mode contrast must be verified in the live app per the "verify against live app" rule.
