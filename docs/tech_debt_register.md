# Technical Debt Register

This document records intentional technical tradeoffs made in this project. Each entry explains what was chosen, what was not chosen, and why the tradeoff is acceptable given the project scope.

**Project scope reminder:** Internal academic tool used by TED University EEE department. ~10–30 jurors per event, 2–3 events per year.

---

## TD-01 — State-Based Routing (No React Router)

**What we have:** `App.jsx` manages `page` state (`"home" | "jury" | "admin"`). Navigation is a state change.

**What we don't have:** URL-based routing, browser back/forward support, bookmarkable admin tabs, deep links.

**Why acceptable:**
- Jurors use a shared device in a linear flow — they never need to bookmark or share a URL
- Admin operates from one device per session — no need to link to a specific tab
- Eliminates a dependency and reduces bundle size
- Avoids routing conflicts in a multi-step form flow

**Risk:** If the app ever needs to support multiple concurrent admin sessions or direct URL sharing, this will require a full routing migration.

---

## TD-02 — Stateless Admin Password per RPC

**What we have:** Admin password is passed as a parameter (`p_admin_password`) to every protected RPC call.

**What we don't have:** Server-side sessions, JWT tokens, OAuth, or any session management.

**Why acceptable:**
- Application is used internally by a known administrator
- Password is stored in `useRef` (not React state) — not exposed in DevTools or serialized
- Password is stored in `localStorage` only for same-device convenience — cleared on logout
- RPC functions validate the password against a bcrypt hash on every call (server-side)
- The threat model for an internal EEE department tool does not require session-based auth

**Risk:** If the password is compromised (e.g., shoulder-surfing, clipboard snoop), there is no session revocation mechanism.

---

## TD-03 — 4-Digit Juror PIN

**What we have:** Jurors authenticate with a 4-digit numeric PIN (0000–9999, 10,000 possible values).

**What we don't have:** Longer PINs, password-based juror auth, or email-based OTP.

**Why acceptable:**
- Juror PINs are generated by the system (CSPRNG via `gen_random_bytes`) — not user-chosen weak values
- Rate limiting is enforced at the DB layer (`locked_until` timestamp after N failed attempts)
- The attack surface is low: jurors are invited faculty members operating on a trusted network
- 6-digit PINs increase friction and errors during a fast-paced evaluation event

**Risk:** A determined attacker with network access could enumerate PINs across accounts if rate limiting is bypassed. Not a realistic threat for this context.

---

## TD-04 — No RPC Response Caching

**What we have:** Data is re-fetched from Supabase on every tab switch. No client-side cache.

**What we don't have:** React Query, SWR, or any cache invalidation logic.

**Why acceptable:**
- On evaluation day, scores are actively changing every few minutes — stale data is a real risk
- Data volumes are small (< 30 jurors, < 20 projects per semester) — fetches are fast
- Adding cache invalidation logic would add complexity without meaningful performance benefit

**Planned mitigation:** Add a "Last updated" timestamp and manual refresh button to the Overview tab (see `docs/implementation_plan.md` Phase 2.1).

---

## TD-05 — Admin Prop Drilling

**What we have:** `adminPass` and `onAuthError` are passed as props from `App.jsx` → `AdminPanel.jsx` → child tabs.

**What we don't have:** React Context, Zustand, or any state management library.

**Why acceptable:**
- The component hierarchy is shallow — prop drilling is at most 2–3 levels deep
- Admin password should not be in a context that could be accidentally subscribed to by unrelated components
- Adding a context/library for a flat hierarchy would be over-engineering

**Risk:** If the admin panel grows significantly in depth (e.g., deeply nested modals), this will need to be refactored.

---

## TD-06 — Limited Mobile Support for Admin

**What we have:** Admin panel is optimized for desktop/laptop use.

**What we don't have:** Fully responsive admin UI, touch-optimized score grid, DnD on mobile.

**Why acceptable:**
- The admin operates from a department computer during event setup
- The score grid uses react-window virtualization — complex to make fully touch-responsive
- Juror evaluation flow IS mobile-friendly (a separate, simpler UI)

---

## TD-07 — Single Admin User

**What we have:** One global admin password. One admin at a time.

**What we don't have:** Multi-admin, role-based access control, admin audit trails (beyond Supabase logs).

**Why acceptable:**
- The department has one designated coordinator managing poster day
- Adding RBAC for a single-person operation is pure overhead

---

## TD-08 — `src/Charts.jsx` Compatibility Shim

**What we have:** `src/Charts.jsx` re-exports everything from `src/charts/index.js`. It exists to avoid breaking any import that referenced the old monolith path.

**What we don't have:** A single import path — callers should prefer `./charts/ComponentName`.

**Resolution:** This shim can be removed once all direct imports of `src/Charts.jsx` are confirmed absent. Safe to remove — it is a pure re-export.
